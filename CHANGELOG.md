## Changelog

All notable changes to this project are documented here in a high‑level, date‑based format.

### 2026-03-15

- **Multi-document chat**: Fixed chat so that when multiple documents are selected (e.g. three pathology reports), all selected documents are sent to the API and included in context with full detail. (1) Frontend: Chat page now passes all selected IDs (`documentIds` array) to ChatWindow, which sends both `document_id` (first) and `document_ids` (all) to the backend. (2) Backend: Fixed if/elif logic to prioritize `document_ids` over single `document_id` so all IDs are processed. (3) Chunk retrieval: Increased minimum chunks per document from 5 to 10 (MIN_PER_DOC=10) in multi-doc retrieval mode; raised full-document threshold from 50 to 100 chunks so 3 typical reports (60-70 chunks total) receive full content instead of sparse retrieval. (4) Prompt: Added "CRITICAL FOR MULTIPLE DOCUMENTS" instruction requiring LLM to interpret each document individually with full detail (all findings, diagnoses, results) and compare/relate them afterward; prevents superficial summaries.
- **Generic retrieval improvements for holistic queries**: Improved chat retrieval so broad questions (e.g. “interpret this document”, “summarise the report”) get sufficient context without requiring users to ask targeted questions. (1) Adaptive top_k: retrieval size now increases with document chunk count (capped at 25) so longer documents send more chunks to the LLM. (2) Document-theme retrieval: when a document has more than one chunk, a second hybrid search using the first chunk’s text snippet is run and results are merged with RRF, so both query-relevant and document-representative chunks are included. (3) Last-chunk inclusion: for documents with at least six chunks, the last chunk is always included so conclusion/summary content is available. (4) Chat prompt: added an instruction for the LLM to use the full context and synthesise across sections when the user asks to interpret, summarise, or explain the document as a whole. All logic is generic (no language or document-type hardcoding). New `merge_rrf_result_lists` in vector_store and `_effective_top_k` helper; unit tests in `tests/test_vector_store.py`.

### 2026-03-12

- **Vector search async fix**: Fixed chunk search and chat retrieval timeout issues by wrapping `VectorStore.search()` and `VectorStore.hybrid_search()` (FAISS + ML model operations) with `asyncio.to_thread()` in chunk and chat routers. Blocking synchronous operations (FAISS index search, SentenceTransformer model loading) were freezing FastAPI's event loop for 95+ seconds, causing frontend "Network Error". Now completes in ~3 seconds by offloading to thread pool while keeping event loop responsive.
- **Hybrid retrieval (BM25 + FAISS)**: Implemented professional-grade hybrid search combining keyword-based BM25 with semantic FAISS using Reciprocal Rank Fusion (RRF). Created BM25Retriever service with rank-bm25 library, added hybrid_search method to VectorStore with configurable alpha/beta weights, integrated into document pipeline and chat router. Dramatically improved retrieval quality for contract queries (e.g., "Çalışanın Unvanı") by combining exact term matching with semantic similarity.
- **Table and field extraction**: Added pdfplumber-based structured data extraction for legal/contract documents. Created TableFieldExtractor service to detect tables and key-value pairs from PDFs. Extended Chunk model with chunk_type (clause/field), field_label, field_value, and field_type metadata columns. Integrated into PDF ingester to extract and index structured fields alongside regular text chunks. Key-value pairs (e.g., "Employee Title: Software Engineer") now indexed as separate FieldChunks for better retrieval.
- **Enhanced semantic chunking**: Upgraded StructuredDocumentChunker with LangChain's SemanticChunker using gradient threshold for embedding-based splitting. Hybrid approach: structural phase (numbered sections) + semantic phase (similarity-based splitting). Large sections now split by semantic coherence instead of arbitrary paragraph boundaries. Preserves document structure while respecting semantic flow. Fallback chain: semantic → paragraph → raw.
- **Prompt hardening**: Strengthened chat LLM prompt with anti-hallucination rules: (1) Only answer if exact info in context, (2) If not found, respond with "I cannot find that information in the document", (3) Never invent or extrapolate, (4) Never merge different placeholders, (5) Copy field labels exactly as they appear in context. Prevents fabricated answers for contract queries.
- **Dependencies**: Added rank-bm25==0.2.2, pdfplumber==0.11.5, langchain-text-splitters==1.1.0, langchain-experimental==0.3.6 to requirements.txt.
- **PII detection improvements**: Upgraded all language NER models to state-of-the-art XLM-RoBERTa based models (Davlan/xlm-roberta-base-wikiann-ner for 20 languages, akdeniz27/xlm-roberta-base-turkish-ner for Turkish, Babelscape/wikineural-multilingual-ner for European languages). Improved Ollama Layer 3 prompt to capture exact lowercase person names in context (e.g., family member names). Enhanced blocklist logic to include all entity tokens regardless of casing, ensuring residual mentions are masked. Fixed Ollama span matching to be case-insensitive and increased Ollama output token limit to 2048 for complete JSON responses.
- **KVKK ruleset (6698)**: Expanded entity types to align with Madde 3(d) (kişisel veri: her türlü bilgi) and Madde 6 (özel nitelikli kişisel veriler, 11 kategori). Added: SOCIAL_SECURITY_NUMBER, TAX_ID, COORDINATES, IP_ADDRESS, COOKIE_ID, DEVICE_ID, DNA_PROFILE, MEDICATION, CLINICAL_NOTE, SEXUAL_ORIENTATION, FINANCIAL_ACCOUNT, CREDIT_CARD_NUMBER, BANK_ACCOUNT_NUMBER. REGULATION_ENTITY_SOURCES.md updated with exact article wording, Kurum examples, and full Madde 6 list; recognizer pack docstring updated.
- **Regulation rulesets (GDPR, UK GDPR, CCPA/CPRA, PIPEDA)**: Aligned built-in regulation entity types with official legal texts: GDPR Art. 4(1), Art. 9(1), Recital 30 (added FIRST_NAME, LAST_NAME, ETHNICITY, MAC_ADDRESS, STREET_ADDRESS, CITY, COORDINATES, CLINICAL_NOTE, PASSPORT_NUMBER); UK GDPR same set; CCPA/CPRA § 1798.140 (added ETHNICITY, SOCIAL_SECURITY_NUMBER, FIRST_NAME, LAST_NAME, STREET_ADDRESS, CITY; CPRA added health-related types); PIPEDA s. 2(1) and OPC guidance (added FIRST_NAME, LAST_NAME, STREET_ADDRESS, CITY, MEDICATION). Descriptions and official URLs updated; REGULATION_ENTITY_SOURCES.md updated with exact article/recital wording and PIPEDA subsection.
- **Regulation entity sources doc and rule**: Add backend/docs/REGULATION_ENTITY_SOURCES.md; rule in .cursor requires updating the doc when changing built-in regulation entity types. Seed adds use_ollama_layer to AppSettings defaults.
- **Sanitizer and PII pipeline**: Ollama PII layer wrapped in try/except so failures do not break sanitization; NER uses language-aware confidence threshold (lower for non-English PERSON_NAME/LOCATION); anonymization blocklist adds single-token entities so residual mentions are redacted; NonPiiFilter logic fixed to keep spans when no rule matches; sanitizer_alias_layer prompt made generic (person, location, alias).
- **Chat query sanitization**: User message is now sanitized before retrieval and LLM call (same pipeline as document text).
- **E2E test and ingestion errors**: E2E Turkish PII test no longer skips: uses Presidio-detectable email, language and anon-map mocks, and LLM mock with `on_cloud_failure`; fixture sets `TLDEXTRACT_CACHE` to tmp path. Document upload 500 response now includes `ingestion_error` in detail for easier debugging.
- **NER model overrides**: NER Models settings tab now supports per-language overrides: edit the HuggingFace model ID for any language, restore default per row (persists immediately), and save overrides to the database. Backend stores overrides in `app_settings.ner_model_overrides` and uses them in the sanitizer pipeline and chat.
- **Error logging and Error Log UI**: Centralized error logging for backend and frontend: new `ErrorLog` model and table, `error_logger` service, global exception handler, and `POST /api/error-logs/frontend` for client-reported errors. New Error Logs page under Settings lists, filters, and clears logs with optional stack-trace detail; frontend global error boundary and runtime error listeners report errors to the backend. Sidebar shows error count badge; handled backend errors (e.g. LLM failures, test-llm fallback) are logged; Test LLM returns 200 with `ok: false` when cloud fails and Ollama fallback is used.
- **Document preview**: Copy button shows "Copied" state for user feedback.
- **Chat**: Badge under assistant message when the answer was produced by local Ollama fallback (cloud unavailable).
- **Docs**: README (EN/TR) add Changelog and License links in header.
- **Changelog and rules**: Split changelog entries by commit date (2026-03-11 vs 2026-03-12). Changelog rule updated to require always verifying today’s date (`date +%Y-%m-%d`) and assigning entries to the correct day via `git log --date=short`.
- **Backend dependencies**: Bumped `langchain-experimental` from `0.3.6` to `0.4.1` in `backend/requirements.txt` to restore backend CI compatibility with Python 3.13 on GitHub Actions and keep dependencies aligned with the latest stable version.

- **Recognizer regex and E2E test**: Fix Presidio regex patterns in GDPR, HIPAA, and KVKK packs: use correct word boundary `\b` and escapes so email/phone/IP patterns match. Set TLDEXTRACT_CACHE in conftest before imports to avoid permission errors in E2E tests.
- **Backend tests and CI**: Fix E2E Turkish PII test in CI by preventing PDF OCR from replacing the text layer (patch `PdfIngester._run_ocr_on_page` in fixture so Presidio detects emails). Replace deprecated `datetime.utcnow` with timezone-aware `datetime.now(timezone.utc)` in models (document, regulation, error_log, spreadsheet_schema) and tests. Add `pytest.ini` with filterwarnings to suppress EasyOCR, Presidio, torch, and Swig deprecation warnings in test runs.
