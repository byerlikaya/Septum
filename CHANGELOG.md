### 2026-04-13

- **Stop PHONE and validator drops from silently eating NATIONAL_ID**: Two independent bugs conspired to produce the same symptom on a 26-document multilingual fixture — "national IDs don't get detected, they look like phone numbers". Root causes: (1) `ExtendedPhoneRecognizer`'s pattern started with `(?:\\+?\\d{1,3}\\s*)?` — the `+` was optional, so the optional prefix group greedily consumed 1-3 digits and turned any 11-13 digit identifier into a phone match. Japanese My Number `701040108923`, Saudi Iqama `101048293`, South African ID `482019384`, typed TCKN and similar values were all silently mis-classified as PHONE_NUMBER. Dedup then preferred PHONE (score 0.80) over the regulation pack's NATIONAL_ID detector (format-only 0.55-0.60) at the same span offsets, because `_HIGH_PRIORITY_ENTITY_TYPES` contained PHONE but the ID families CPF / SSN / TAX_ID / PASSPORT / DRIVERS_LICENSE / MEDICAL_RECORD_NUMBER / HEALTH_INSURANCE_ID / LICENSE_PLATE lived in the low-priority bucket, so they lost automatically to any overlapping PHONE span. The GDPR and KVKK packs also shipped their own `_phone_recognizer` at `\\b\\+?[0-9]{2,}\\b` / `\\b\\+?[0-9]{7,15}\\b` — strict subsets of the baseline regex, but triggered on every 7-15 digit sequence with no context and no separator requirement, compounding the false-positive problem. (2) `ValidatedPatternRecognizer.analyze` dropped any span whose algorithmic validator returned False — so a synthetic TCKN like `58291047382` in `2_hasta_kayit_formu.pdf`, or a test CPF like `473.829.104-56`, or a typo'd Aadhaar simply never reached the detection pipeline. The checksum gate silently leaked every test fixture and every human-entered identifier with a single-digit typo. **Fix (a) — phone regex tightened**: `ExtendedPhoneRecognizer._pattern` now matches one of: an explicit `+` international prefix followed by a flexible body, or at least three separator-delimited digit groups (space / dash / dot), with an optional parenthesised area code. A length sanity check caps the match at 9–15 digits so the prefix can never eat the first digits of a long identifier. **Fix (b) — dedup priority**: `span_processing.deduplicate_spans` gained `_ENTITY_TYPE_PRIORITY` as a tiebreaker inside `_dedup_simple`, ranking IBAN / CREDIT_CARD_NUMBER (100) and CPF / NATIONAL_ID / SOCIAL_SECURITY_NUMBER (95) above PASSPORT_NUMBER / MEDICAL_RECORD_NUMBER / HEALTH_INSURANCE_ID (90), TAX_ID (85), DRIVERS_LICENSE (80), LICENSE_PLATE (75), and finally PHONE_NUMBER (70). `_HIGH_PRIORITY_ENTITY_TYPES` was expanded to include every one of these identifier types so they all fight in the same dedup pool and the priority tiebreaker actually applies. `PIISanitizer._register_custom_recognizers` dropped the duplicate phone recognizers from the GDPR and KVKK packs (the baseline detector already covers every real format the packs hit). **Fix (c) — format-only fallback for checksum-failing IDs**: new `RegexPatternConfig.fallback_score` parameter. When set, `ValidatedPatternRecognizer.analyze` keeps a failed-validation match alive at the reduced fallback score instead of dropping it. Wired into both KVKK national-ID recognizers (`fallback_score=0.55`/`0.5`), the DPDP Aadhaar recognizer (`0.55`), and — via direct implementation in their custom classes — the baseline `SSNRecognizer` (context-gated, keeps failed matches at `0.55`) and `CPFRecognizer` (now context-gated on the `cpf` keyword, same `0.55` fallback). **Fix (d) — POPIA spaced SA ID format**: the 13-digit SA national-ID pattern was `\\b\\d{13}\\b` and missed the common printed form `880317 5432 087`; rewritten as `\\b\\d{6}[\\s\\-]?\\d{4}[\\s\\-]?\\d{3}\\b` to accept single-space / single-dash grouping. **Impact on the 26-doc fixture**: cumulative detection count 1098 → 1311 (+19%). Documents that previously had 0 NATIONAL_ID are now healthy: APPI JP native 0 → 9, PDPL SA 0 → 9, POPIA ZA 0 → 6, APPI JP 0 → 7, DPDP IN 0 → 6, PIPL CN native 0 → 6, 2_hasta_kayit_formu 0 → 1 (synthetic TCKN now masked at fallback score). Test coverage: 3 new tests (`test_phone_regex_rejects_bare_long_digit_sequences`, `test_national_id_wins_over_phone_on_equal_span`, `test_invalid_tckn_with_prefix_is_still_masked_at_fallback_score`). Full suite: 265 backend tests pass.
- **Unblock setup wizard with bootstrap-mode auth relaxation**: The earlier RBAC commit locked every admin-only endpoint behind `require_role("admin")`, but the setup wizard runs *before* the first admin is created and has no JWT to present. Result: opening a freshly deployed Septum produced `401 Unauthorized` on `GET /api/settings/ollama-models`, `PATCH /api/settings`, `POST /api/settings/test-llm`, `POST /api/settings/test-local-models`, `GET /api/settings/ollama-probe`, `POST /api/settings/ollama-pull`, `GET /api/regulations` and `PATCH /api/regulations/{id}/activate` — effectively every step of the wizard except the final `/api/auth/register`. Two new dependencies in `backend/app/utils/auth_dependency.py` — `require_admin_or_bootstrap` and `require_user_or_bootstrap` — relax admin / authenticated enforcement while the system is still in first-run bootstrap, defined as `users` table empty **and** `AppSettings.setup_completed` still `False`. The moment the first admin is created (via `/api/auth/register`) or the wizard flips `setup_completed` to `True`, the dependencies snap back to their original strict behaviour — equivalent to `require_role("admin")` / `get_current_user`. Applied to all eight wizard-facing endpoints (`settings.py` × 6, `regulations.py` × 2). A new `TestBootstrapModeAccess` class in `tests/test_rbac.py` pins the bootstrap-open behaviour, and the existing `TestUnauthenticatedAccess` tests all gained a `_bootstrap_admin(client)` setup call so they continue to prove the gate slams shut once an admin exists. Full suite: 263 backend tests pass.
- **GDPR and LGPD packs gain context-gated country-specific national-ID recognizers**: Extending the structural fixes one layer deeper, the GDPR recognizer pack now carries format-descriptive detectors for the main EU national-ID families it previously missed: German Personalausweis (letter + 8 digits, gated on `personalausweis`/`ausweisnummer`/`id-karte`), German 11-digit Steuer-ID (spaced or compact, gated on `steuer-id`/`steueridentifikationsnummer`/`identifikationsnummer`), German Rentenversicherung (`NN NNNNNN L NNN` format, gated on `rentenversicherungs-nr`/`sozialversicherungs-nr`/`rv-nr`/`sv-nr`), Spanish DNI (8 digits + check letter in either compact `28741095Z` or dotted `50.432.187-K` form, gated on `dni`/`documento nacional`), Spanish NIE (`[XYZ]\d{7}[A-Z]`, gated on `nie`/`número de identificación de extranjero`), Spanish Seguridad Social (11-12 digits with flexible grouping, gated on `seguridad social`/`nss`), French NIR (15-digit grouped form, gated on `sécurité sociale`/`nir`), and French SIREN (`NNN NNN NNN`, gated on `siren`/`siret`). Every recognizer uses `narrow_to_group=1` so only the identifier value — not the context keyword and delimiter — lands in the reported entity span, and the context keyword tuples sit behind a module-level `# FUTURE: move to DB` comment. Function names are deliberately format-descriptive (`_letter_prefix_8digit_national_id_recognizer`, `_eleven_digit_tax_identifier_recognizer`, `_eight_digit_plus_letter_national_id_recognizer`, etc.) rather than country-named so the zero-tolerance generic rule stays clean. The LGPD pack similarly gains a `_civil_identity_document_recognizer` for the Brazilian RG that matches both `45.238.917-4 SSP/SP` (dotted + issuing-authority suffix) and `12345678X` (compact). Two new tests lock the behaviour down: `test_gdpr_pack_country_specific_national_ids` walks all 9 GDPR cases with a GDPR-only policy (no baseline NATIONAL_ID recognizer, to prove the pack carries detection end-to-end), and `test_lgpd_pack_civil_identity_document` covers the LGPD pack. Final reprocess on the 11-document multilingual fixture: 378 → 620 (+64%) vs the original ingest, zero regressions. Full suite: 259 backend tests pass.
- **Broaden IBAN / DOB / TAX_ID detection across locales**: Three blind spots were visible once a multilingual fixture (tr/de/es/fr/en/pt) was reprocessed against the current sanitizer. (1) `ValidatedIBANRecognizer` only matched the contiguous form and only emitted on MOD 97-10 valid values, so paper-format IBANs (`DE89 2004 1010 0540 1420 00`) and synthetic/test IBANs were silently dropped. (2) `DateOfBirthRecognizer` only recognized purely numeric date formats, so German "22. Juli 1991", French "09 février 1987", Spanish "01 de octubre de 2024", Portuguese "12 de março de 1983", and English "August 15, 1980" all slipped past the DOB layer into plain text. (3) `TaxIDRecognizer._pattern` was `\b\d{2}[\-]?\d{7,10}\b`, which failed on the German slash-separated Steuernummer `27/431/07892`, the EU VAT letter-prefix form `DE298431076` / `FR 47 834291076`, the dotted Brazilian Inscrição Estadual `123.456.789.112`, the Spanish CIF `B-86432178`, and the 11-digit spaced German Steuer-ID `77 523 164 890`. Fix: `ValidatedIBANRecognizer` now matches both spaced and contiguous IBANs via `\b[A-Za-z]{2}\d{2}[\s0-9A-Za-z]{11,32}`, post-strips whitespace, validates length 15–34, and emits IBAN at `0.96` when the checksum passes and `0.65` as a format-only fallback when it fails — matching the "over-detection is preferred" philosophy already used by `CreditCardNumberRecognizer`. `DateOfBirthRecognizer` gains a `_MONTH_NAMES` tuple covering en/de/fr/es/pt/it/tr and two new regex branches for `day + month-name + year` and `month-name + day + year`, both still gated by the existing DOB context keyword window. `TaxIDRecognizer._pattern` is rewritten as a six-family alternation covering EU VAT prefixes, single-letter CIF prefixes, slash-separated Steuernummer, dotted Inscrição Estadual, 11-digit spaced Steuer-ID, and the original compact form; the context keyword list is extended with `ust[\-\s]?id`, `USt[\-\s]?IdNr`, `CIF`, `NIE`, `SIREN`, `SIRET`, `tva intracommunautaire`, `partita iva`, `inscrição estadual`, `CNPJ`, and `CPF`. `PIISanitizer._expand_with_aliases` is extended with `_PARENT_TYPE_COVERAGE` expansion so a regulation that declares `BANK_ACCOUNT_NUMBER` no longer silently hides the IBAN recognizer from Presidio's entity filter (same for `FIRST_NAME`/`LAST_NAME` → `PERSON_NAME`). Four new tests in `tests/test_sanitizer.py` lock down the behaviour: `test_iban_space_grouped_valid`, `test_iban_format_only_fallback_for_synthetic`, `test_date_of_birth_month_name_multilingual` (en/de/fr/es/pt), and `test_tax_id_broadened_patterns` (6 language variants). Full suite: 257 backend tests pass.
- **Detection refresh script so an installed corpus picks up new recognizers without re-ingesting PDFs**: Real documents already loaded into a Septum instance carry a frozen snapshot of whatever the sanitizer could do at upload time — if a later release ships a broader date-of-birth regex or a new national-ID pack, existing `entity_detections` rows and on-disk `anon_maps/*.enc` files do not pick that up, and a chat query against an old document will miss the newly-detectable PII. The only remedy used to be a full re-upload, which drops the raw chunk text, re-runs PDF extraction, re-builds embeddings, and re-chunks — expensive and unnecessary. New `backend/scripts/reprocess_entity_detections.py` does the narrow thing: reads every document from the DB, re-sanitizes each chunk's already-stored text through the current `PIISanitizer` (Presidio + NER, no Ollama — matching the ingestion pipeline), rewrites `entity_detections`, overwrites the encrypted `anon_maps/{id}.enc` payload, and updates `documents.entity_count`. Chunks, vector indexes, and BM25 indexes are left untouched — there is nothing to rebuild because the raw text is unchanged. It respects each document's own `active_regulation_ids` so reprocess cannot silently widen the user's policy, uses `SEPTUM_CONFIG_PATH` to resolve the bootstrap config (needed for the AES-256-GCM key that protects the anon_map files), and caches a separate `PIISanitizer` per regulation-set so NER model loading happens at most once per distinct combination. On the 11-document multilingual fixture (tr/de/es/fr/en/pt) the refresh lifted the cumulative detection count from 378 at the original ingest to 620 after Phase 1 + 2, a +64% uplift with zero regressions.
- **Every built-in regulation now ships a first-class recognizer pack**: Septum advertised 17 built-in regulations in `backend/app/seeds/regulations.py`, but only GDPR, HIPAA and KVKK had a `recognizers/{reg_id}/recognizers.py` module — the other 14 quietly fell through to the regulation-agnostic baseline, so activating LGPD, CCPA, CPRA, UK GDPR, PIPEDA, PDPA TH/SG, APPI, PIPL, POPIA, DPDP, PDPL SA, NZPA or Australia PA gave the user the same detection set as GDPR. Worse, `AadhaarValidator` in `backend/app/services/national_ids/aadhaar.py` was orphaned — the Verhoeff checksum was implemented but never wired into any recognizer, so real Aadhaar numbers in DPDP documents were silently missed. Fix: created 14 new recognizer packs, each with a `get_recognizers()` entry point, a module docstring pointing to `backend/docs/REGULATION_ENTITY_SOURCES.md`, and at least one region-specific structural or checksum-validated detector. DPDP wires `AadhaarValidator.validate` through `ValidatedPatternRecognizer` for 12-digit IDs formatted `NNNN NNNN NNNN`; LGPD adds a format-only CNPJ detector; UK GDPR adds a NINO (`AB123456C`) detector; PIPEDA adds the Canadian SIN; PDPA Thailand adds the 13-digit grouped national ID; PDPA Singapore adds the `[STFGM]\d{7}[A-Z]` NRIC/FIN; APPI adds 12-digit My Number; PIPL adds the 18-character Resident ID; POPIA adds the 13-digit SA ID; PDPL Saudi Arabia adds the 10-digit Iqama/National ID; NZPA adds the 8-9 digit IRD; Australia PA adds TFN and Medicare card; CCPA and CPRA add California driver's licence plus the common email fallback. New `tests/test_recognizer_packs.py` parametrises over `builtin_regulations()` and asserts every regulation ID exposes a loadable `get_recognizers()` that returns at least one `EntityRecognizer`, then for each new pack runs a representative sample string end-to-end through `PIISanitizer` with a matching `ComposedPolicy` and asserts the expected placeholder appears — 31 new tests, all green. `README.md` / `README.tr.md` Multi-Regulation bullet updated to explicitly advertise the region-specific national ID detectors (TCKN, Aadhaar Verhoeff, NRIC/FIN, Resident ID, NINO, CNPJ, My Number, and more).

- **Document preview authenticates via blob URL instead of a naked `<iframe src>`**: Opening the Document Preview on a freshly-authenticated admin session showed `{"detail":"Not authenticated"}` in the "Original document" pane. Root cause: `DocumentPreview.tsx` was mounting the raw file with `<iframe src={`${baseURL}/api/documents/${id}/raw`}>` — native HTML elements like `<iframe>`, `<img>`, and `<audio>` cannot carry the axios interceptor's `Authorization: Bearer …` header, so the browser hit the `/raw` endpoint unauthenticated and the backend (now gated by `require_role`) correctly returned 401. Fixed by fetching the raw file through the existing typed axios client as a `Blob`, wrapping it with `URL.createObjectURL`, and passing the resulting object URL to `<iframe>` / `<img>` / `<audio>` instead. The blob URL is created inside a `useEffect` keyed on `open` + `document.id` and revoked on unmount / document change via `URL.revokeObjectURL` so blob references do not leak as the user flips between documents. Added a `rawLoadError` state so auth failures or unreachable endpoints surface an inline error panel instead of an invisible broken frame. Removed the no-longer-needed `baseURL` / `getAuthToken` imports. Works identically for PDFs (with `#toolbar=1&page=N` hash anchors), rendered images, and streamed audio.
- **Baseline sanitizer is now regulation-agnostic; KVKK-specific NATIONAL_ID detection moved into the KVKK pack**: `backend/app/services/sanitizer.py::ValidatedNationalIDRecognizer` defaulted its validator to `TCKNValidator`, which meant every `PIISanitizer` — even with GDPR, HIPAA or DPDP active — was filtering 11-digit NATIONAL_ID candidates through a Turkish checksum. 11-digit numbers are ambiguous across many jurisdictions (Thai ID, Ukrainian ID, Colombian ID, Chinese resident ID fragments…), so silently defaulting to TCKN is structurally wrong. Fix: deleted the baseline class entirely and removed it from `_register_custom_recognizers`; country-specific checksum detection now lives only in regulation packs, starting with `kvkk/recognizers.py::_validated_national_id_recognizer` and `_contextual_national_id_recognizer` (both keyed off `TCKNValidator`). Also cleaned `SSNRecognizer`'s context regex — removed `sosyal güvenlik|sgk` so the US-scoped Social Security Number detector no longer silently scoops up Turkish SGK mentions (which are a different identifier entirely). `CPFRecognizer` docstring and sanitizer `_titlecase_upper_segments` example both lost their locale-specific wording. The multilingual keyword lists in `PassportNumberRecognizer`, `DriversLicenseRecognizer`, `TaxIDRecognizer` and `LicensePlateRecognizer` stayed in place but got explicit `# FUTURE: move context keywords to DB` comments so the intent is recorded. The `kvkk/recognizers.py` module docstring was trimmed to a generic one-liner pointing at `REGULATION_ENTITY_SOURCES.md`, `_TCKN_VALIDATOR` was renamed to `_NATIONAL_ID_VALIDATOR`, and the contextual recognizer now sources its keywords from a `_NATIONAL_ID_CONTEXT_KEYWORDS` tuple (also annotated `# FUTURE: move to DB`) with `\b` word-boundary anchors, removing the inline `kimlik|tc` alternation that mixed Turkish terms into the production pattern. `tests/test_sanitizer.py` picks up a new `_kvkk_policy()` helper so the existing TR-language fixtures keep passing now that the baseline no longer ships a TCKN detector. Full suite: 253 backend tests pass (222 pre-existing + 31 new pack smoke tests).
- **National ID detection now narrows the span to the digits and runs the TCKN checksum**: Two bugs in the KVKK recognizer pack conspired to produce misleading results like `No: 61504839271 → NATIONAL_ID @ 60%`. First, `kvkk_national_id_context` in `backend/app/services/recognizers/kvkk/recognizers.py` used a regex of the form `(?i)(?:…|no)[^\d]{0,16}\d{8,12}` with no capturing group, so the reported entity span included the keyword and delimiter (`No: `) on top of the digits — making the displayed match look noisy and eating tokens in the approval modal. Second, `ValidatedPatternRecognizer` in `backend/app/services/recognizers/base_recognizer.py` was misleadingly named: it validated that the *regex* compiled, but it never called the algorithmic validators under `backend/app/services/national_ids/` (`TCKNValidator`, `CPFValidator`, `IBANValidator`, `SSNValidator`, `AadhaarValidator`), so every 11-digit string that matched the pattern got full score regardless of whether it passed the two-check-digit TCKN algorithm. `61504839271` was exactly this case — a synthetic AI-generated test number whose computed check digits (2, 5) did not match the printed digits (7, 1). Fix: `RegexPatternConfig` gains an optional `narrow_to_group: Optional[int]` that tells the recognizer to report only a specific capture group as the entity span; `ValidatedPatternRecognizer` gains an optional `algorithmic_validator: Callable[[str], bool]` hook, a `validate_result` override that runs it via Presidio's standard validation path, and an `analyze` override that applies the capture-group narrowing and filters out algorithmically invalid matches entirely (instead of leaving them at `score=0`, which some downstream filters would still surface). The two KVKK NATIONAL_ID recognizers now both pass `TCKNValidator().validate` as their algorithmic validator; the context recognizer also uses `narrow_to_group=1` with a fresh pattern that captures just the digits. Also tightened `deduplicate_spans` in `backend/app/services/span_processing.py` to use an explicit overlap check across all already-chosen spans instead of a single `current_end` tracker, which only works when sorted by start — the sort key itself stays the same, so existing entity-type precedence is preserved. GDPR and HIPAA packs are unaffected: their entities (email, IPv4, phone, MRN, insurance ID) have no standard checksums and their patterns do not use keyword-gap matching. Two new tests in `tests/test_sanitizer.py` lock down the behaviour: `Kimlik No: 10000000078` is detected with the `Kimlik No:` prefix surviving in the sanitized text (only the digits become a placeholder), and `No: 61504839271` is never masked as `NATIONAL_ID` (the checksum fails). Presidio benchmark recall climbed from 94.4% to 95.7% (F1 97.1% → 97.8%, precision still 100%) as the narrowed NATIONAL_ID span now matches the planted entity exactly.
- **Role-based access control enforced across every router**: Until this change the `role` column on the `users` table was cosmetic — only `/api/users/*` actually honoured it, and the rest of the API was either fully open or used an `get_optional_user` backward-compat shim that made anonymous requests indistinguishable from authenticated ones. Now every router declares a concrete auth dependency: read paths (`GET /api/documents`, `GET /api/chunks`, `GET /api/settings`, `GET /api/regulations`, approval flow, chat) require `get_current_user`; write paths split by role — documents upload/delete/reprocess/language/schema and chunks PATCH/DELETE are `require_role("admin", "editor")`, while settings PATCH, LLM/Ollama admin, regulation mutations, audit, error-logs, text-normalization, chat debug and post-install `/api/setup/infrastructure` are `require_role("admin")`. The frontend error-reporting `POST /api/error-logs/frontend` keeps optional auth so pre-login failures still land in the log. Documents are now explicitly shared org-wide: `list_documents` no longer filters by `user_id`, and `Document.user_id` is retained purely as audit attribution for who uploaded what. Chat sessions, in contrast, stay per-user — a new `_get_owned_session_or_404` helper rejects cross-user GET/PATCH/DELETE/export/messages requests with 404 (not 403) so session ids cannot be probed for existence. Test coverage: new `tests/test_rbac.py` (27 tests) walks the full matrix — unauthenticated rejection on every surface, editor-can-read-settings / editor-cannot-mutate, viewer-can-chat / viewer-cannot-upload, and cross-user chat session isolation. Existing `test_approval_router.py` and `test_e2e_chat_turkish_pii.py` inject a fake admin via `app.dependency_overrides` so TestClient calls pass the new gates. Full backend suite: 220 tests pass.
- **User management with admin-only CRUD, self change-password, and role-gated navigation**: Septum's authentication system previously exposed only `register`, `login`, and `me` — there was no way to list users, create additional accounts, rotate passwords, or deactivate someone from the UI, and the `role` column on the `users` table was dead code because no endpoint ever enforced it. New backend router `backend/app/routers/users.py` adds `GET/POST /api/users`, `PATCH /api/users/{id}`, `POST /api/users/{id}/reset-password`, and `DELETE /api/users/{id}`, all gated via `require_role("admin")`. The existing self-signup endpoint is now a pure bootstrap path: when the `users` table is empty the first caller is promoted to `admin`, and every subsequent call returns 403 (`Self-registration is disabled`). The setup wizard's final step already hits `/api/auth/register`, so the wizard naturally creates the root administrator without any additional UX changes. Defense-in-depth guards block self-delete / self-deactivate / self-role-change and reject destructive actions against the last remaining active admin. A new `POST /api/auth/change-password` lets the authenticated user rotate their own password with the current password verified, a fresh JWT returned, and the same 8-character minimum applied to all password entry points. Shared helper `_require_strong_password` keeps the policy in one place. Frontend gains a new `/settings/users` page with a `DataTable` of every account (email, role, status, created, actions), reusable `UserFormModal` / `PasswordResetModal` / `ChangePasswordModal` components, sidebar `Users` entry that only renders when the current user's role is `admin`, and a top-bar "Change password" action wired to a modal from `AppShell`. Current user email + role are now shown in the desktop sidebar footer and the top bar. Obsolete standalone `/register` page was deleted (self-signup is no longer a concept once the wizard has run), its i18n keys dropped, the `authRegister` helper removed from `src/lib/api.ts`, and `AUTH_PAGES` / 401 interceptor trimmed to `/login` only. Backend coverage: 29 new tests across `tests/test_auth_router.py` (first-user-admin, subsequent-register-forbidden, min-8 password policy, change-password rotate-token, wrong current rejected) and `tests/test_users_router.py` (admin-only gate for viewer/editor, list/create/get/update/delete happy paths, duplicate email 409, invalid role 422, admin reset-password rotation, self-protection for delete/deactivate/role-change, direct unit test of `_count_active_admins` helper for the last-admin invariant).
